## Rules (non-negotiable)

These rules apply to ALL generated UI code. Violations will be caught in review.

### Tokens

1. **Never use raw values.** Every color, spacing, font-size, shadow, and radius MUST reference a design token via CSS custom property (`var(--token-name)`) or the equivalent framework binding.
2. **Never reference primitive tokens in components.** Components use semantic or component-level tokens only. Primitive tokens (`color.gray.500`, `space.4`) are consumed by semantic tokens, not by UI code.
3. **Token resolution order:** Component token → Semantic token → Primitive token. Always use the most specific available.

### Colors

4. **Text/background contrast:** WCAG 2.1 AA minimum (4.5:1 for normal text, 3:1 for large text and UI elements).
5. **Dark mode:** All color decisions must work in both `light` and `dark` themes. Use semantic tokens — they resolve differently per theme.
6. **Status colors:** `error` = red family, `success` = green family, `warning` = yellow family. Never repurpose these for decorative use.

### Typography

7. **Font sizes:** Use the type scale tokens only (`type.display.*`, `type.heading.*`, `type.body.*`, `type.code.*`). No arbitrary sizes.
8. **Max reading width:** Body text containers must not exceed `65ch`.
9. **Minimum font size:** `type.body.xs` (12px / 0.75rem). Nothing smaller.

### Spacing

10. **Use the spacing scale.** All margins, paddings, and gaps use `space.*` tokens. No arbitrary pixel values.
11. **Consistency rule:** Same relationship = same spacing. If two cards have `space.6` gap, all card grids use `space.6`.

### Components

12. **Use existing components first.** Before creating custom UI, check `COMPONENTS.md` for an existing component that solves the problem.
13. **States are mandatory.** Every interactive element must handle: `default`, `hover`, `focus-visible`, `active`, `disabled`. No exceptions.
14. **Focus ring:** All focusable elements must show `shadow.focus` on `:focus-visible`.

### Accessibility

15. **Semantic HTML first.** Use `<button>`, `<a>`, `<input>`, `<nav>`, `<main>`, etc. Only add ARIA when native semantics are insufficient.
16. **`prefers-reduced-motion`:** All animations must be disabled or reduced when this media query matches.
17. **Touch targets:** Minimum 44×44px for all interactive elements.
18. **Icon-only buttons:** Must have `aria-label`.

### Responsive

19. **Mobile-first.** Base styles = mobile. Use `min-width` media queries to add complexity.
20. **Breakpoints:** Only use the defined breakpoints (`sm: 640px`, `md: 768px`, `lg: 1024px`, `xl: 1280px`, `2xl: 1536px`).

## How to Read Tokens

Tokens file: `tokens/tokens.json`

Structure:
```
{category}.{subcategory}.{name}  →  for primitives
{concept}.{property}.{variant}   →  for semantics
{component}.{variant}.{property}.{state}  →  for component tokens
```

CSS custom property mapping (generated by Style Dictionary):
```
Token:    color.text.primary
CSS:      var(--color-text-primary)

Token:    button.primary.bg.hover  
CSS:      var(--button-primary-bg-hover)

Token:    space.4
CSS:      var(--space-4)
```

## Code Generation Rules

When generating component code, use the project's actual stack:

```tsx
/* ✅ Correct — Tailwind classes + CVA + token variables */
const buttonVariants = cva(
  "inline-flex items-center justify-center rounded-base text-sm font-medium transition-colors focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring disabled:pointer-events-none disabled:opacity-50",
  {
    variants: {
      variant: {
        primary: "bg-primary text-primary-foreground hover:bg-primary-hover",
        secondary: "bg-secondary text-secondary-foreground border border-border",
        danger: "bg-destructive text-destructive-foreground",
      },
    },
    defaultVariants: { variant: "primary" },
  }
);

/* ❌ Wrong — raw CSS values */
.btn-primary {
  background-color: #2563EB;
  padding: 8px 16px;
}

/* ❌ Wrong — primitive token in Tailwind class */
<button className="bg-[var(--color-blue-600)]">
```

Always use:
- `cn()` for class merging (from `@falabella-enablers-genai/ui` or `lib/utils.ts`)
- CVA for variant definitions
- Tailwind utility classes that map to @theme tokens
- `React.forwardRef` and `displayName` on all components

## Drift Detection Protocol

Before generating or reviewing UI code, verify:

1. **Token existence:** Every `var(--token-name)` in code must map to a token in `tokens/tokens.json`. If a token is referenced but doesn't exist, **stop and report**.
2. **No raw values:** Scan for hardcoded hex colors, px spacing, rem font sizes. All must use token references.
3. **No primitive leaks:** Component code must never reference primitive tokens (`--color-gray-500`, `--space-4`). Only semantic/component tokens.
4. **State completeness:** Every interactive element must handle: `default`, `hover`, `focus-visible`, `active`, `disabled`.
5. **Dark mode safety:** All color tokens used must have dark mode overrides in `tokens/semantic/dark.json`.

When drift is detected:
- **Code drift** (code uses wrong/missing token): Fix the code.
- **Design drift** (token is missing or inadequate): Propose an RFC (see `CONTRIBUTING.md`).

## Proposing Design Changes (Agent → Owner)

Agents **cannot** directly modify `tokens/tokens.json`, `docs/DESIGN-SYSTEM.md`, or `docs/DESIGN-OWNER-GUIDE.md`. To propose changes:

1. Identify the gap (missing token, missing component, inconsistency)
2. Create an RFC following the template in `CONTRIBUTING.md`
3. Present it to the Owner for review
4. Only implement after approval

### When to propose vs. when to compose

```
Need a new visual element?
       │
       ▼
Can existing tokens + components solve it?
       │
  YES ──┼── NO
  │           │
  ▼           ▼
Compose    Is it reusable (≥3 places)?
existing        │
tokens     YES ──┼── NO
           │           │
           ▼           ▼
     Propose RFC    Compose with
     for new        existing tokens,
     component      note in review
```

## Sync Awareness

When you detect that `tokens/tokens.json` has changed since your last read:

1. Re-read the token file before generating code
2. Check if any tokens you previously used were renamed or removed
3. Verify dark mode overrides are still complete
4. Check `CHANGELOG.md` for context on what changed and why
